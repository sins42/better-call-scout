Each generator prompt must include these `{state_key}` placeholders (resolved automatically by ADK):
- `{repo_data_json}` — serialized list of RepoData dicts
- `{news_items_json}` — serialized list of NewsItem dicts
- `{rag_chunks_json}` — serialized list of RAGContextChunk dicts
- `{<persona>_critic_output}` — critic feedback from previous iteration (empty string on first pass)
- `{<persona>_draft_output}` — prior draft (empty string on first pass)

Each critic prompt must include:
- `{<persona>_draft_output}` — the hypothesis to challenge

**Implementation notes for prompt content:**

VC_GENERATOR_PROMPT persona (per D-01): decisive and contrarian. Takes strong positions even when evidence is mixed. Produces clear "about to break out" or "overhyped" verdicts. Never hedges. Must focus on: star velocity acceleration, disclosed funding mentions in news, market size signals, competitive landscape dominance. Ends with instruction: "Return a JSON object conforming exactly to AnalystHypothesis. Fields: persona (must be 'vc_analyst'), confidence_score (0.0-1.0), evidence (list of strings), counter_evidence (list of strings), reasoning (string), hypothesis_text (string). No extra keys."

DEV_GENERATOR_PROMPT persona (per D-02): pragmatic engineer. Primary question is "Is this actually production-ready? Would I bet my team on it?" Focuses on DX, community health, ecosystem maturity. Looks at: contributor count, commit frequency, open issue backlog, language ecosystem quality, job posting signals in news. Does not over-index on star counts alone.

JOURNALIST_GENERATOR_PROMPT persona (per D-03): tech media skeptic. Asks "What's the real story vs the hype?" Looks for narrative arc (what story would TechCrunch miss?), incumbent threats, undercovered angles. Evaluates: HN sentiment score, media coverage density, whether mainstream press is late to the story.

Critic prompts (per D-04): Each critic is a separate adversarial devil's advocate. Does NOT share the analyst's perspective. Explicitly instructed to: find flaws in evidence, surface counter-evidence the analyst downweighted, challenge reasoning steps, and identify survivorship bias. The critic output is free-text (no output_schema) — it is consumed as text by the generator on the next iteration.

### Task 2: VC Analyst — Generator, Critic, LoopAgent

**File(s):**
- `src/agents/analysis/vc_analyst.py` (FILL STUB)

**What to build:**

Implement three objects exported from this module: `vc_generator`, `vc_critic`, `vc_analyst_loop`.

```python
from google.adk.agents import LlmAgent, LoopAgent
from src.models.schemas import AnalystHypothesis
from src.agents.analysis._prompts import GEMINI_MODEL, VC_GENERATOR_PROMPT, VC_CRITIC_PROMPT

vc_generator = LlmAgent(
    name="VCAnalystGenerator",
    model=GEMINI_MODEL,
    instruction=VC_GENERATOR_PROMPT,
    output_schema=AnalystHypothesis,
    output_key="vc_draft_output",
)

vc_critic = LlmAgent(
    name="VCCritic",
    model=GEMINI_MODEL,
    instruction=VC_CRITIC_PROMPT,
    output_key="vc_critic_output",
    # No output_schema on critic — free-text challenge is correct here
)

vc_analyst_loop = LoopAgent(
    name="VCAnalystLoop",
    sub_agents=[vc_generator, vc_critic],
    max_iterations=2,
)
```

`__all__ = ["vc_generator", "vc_critic", "vc_analyst_loop"]`

**Implementation notes:**
- Do NOT add `tools=` to `vc_generator` or `vc_critic` — `output_schema` + tools is incompatible in ADK (Pitfall 1 from research)
- `output_key="vc_draft_output"` on the generator means: after each generator run, the `AnalystHypothesis` dict is stored in session state under `"vc_draft_output"`. The VC_GENERATOR_PROMPT uses `{vc_draft_output}` to receive this on iteration 2.
- `max_iterations=2` means the sequence [vc_generator, vc_critic] runs twice: gen(iter1) → critic(iter1) → gen(iter2) → critic(iter2). After the loop, `vc_draft_output` holds the final refined hypothesis.
- Module docstring must describe the agent's persona in one sentence
